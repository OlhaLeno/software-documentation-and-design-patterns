# Лабораторна робота 4: Реалізація патерну "Стратегія" (Strategy Pattern) 🚀

Цей проект демонструє практичне застосування поведінкового шаблону проектування **Strategy (Стратегія)**. 

Програма зчитує великий набір даних про інциденти зі стріляниною в Нью-Йорку (NYPD Shooting Incident Data), перетворює їх у формат JSON та динамічно направляє вивід у різні системи збереження даних (Консоль, бази даних Redis або брокер повідомлень Apache Kafka). Головна особливість — зміна кінцевого сховища відбувається **без жодної зміни в коді програми**, виключно через конфігураційний файл.

---

## Архітектура проекту

Проект побудований за принципами SOLID (зокрема, принцип відкритості/закритості та інверсії залежностей). Логіка розділена на такі ключові компоненти:

* **`interfaces.py` (Абстракція):** Містить базовий клас `IOutputStrategy`. Це "контракт", який гарантує, що будь-яка стратегія виводу матиме метод `output()`.
* **`strategies.py` (Конкретні стратегії):** Тут знаходяться самі алгоритми виводу. 
    * `ConsoleOutputStrategy` — просто друкує дані на екран.
    * `RedisOutputStrategy` — підключається до in-memory бази даних та швидко записує туди масив даних через *pipeline*.
    * `KafkaOutputStrategy` — серіалізує дані та відправляє їх як потік подій у топік брокера Kafka.
* **`processor.py` (Контекст):** Клас `DataProcessor` відповідає за бізнес-логіку (читання CSV та запис у JSON). Він нічого не знає про Redis чи Kafka — він просто передає готові дані тій стратегії, яку йому надали ззовні.
* **`main.py` (Клієнт / Точка входу):** Зчитує налаштування, створює правильну стратегію за допомогою фабричного методу та запускає `DataProcessor`.
* **`.env` (Конфігурація):** Файл змінних середовища. Саме тут відбувається "магія" перемикання стратегій (змінна `OUTPUT_STRATEGY`).
